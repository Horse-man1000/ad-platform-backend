4) IMPORTANT BEHAVIOR FOR FRONTEND
==================================================
- Platform values are normalized to lowercase by backend.
- Token values are masked in responses (security behavior).
- Validation errors and not-found states are returned as JSON with status codes.

Expected status patterns:
- 200/201 success
- 400 invalid input
- 404 not found
- 500 server error

UI should map errors by BOTH:
- HTTP status code
- JSON message field

==================================================
5) DATE/TIME HANDLING
==================================================
- For token expiry fields, send ISO-8601 UTC datetime strings.
- Treat returned datetimes as UTC in data handling.
- Only localize for user display in UI layer.

==================================================
6) RECOMMENDED FRONTEND ARCHITECTURE
==================================================
Create one shared API client layer.

Reason:
- Keeps fetch logic in one place.
- Gives consistent error handling across pages.
- Reduces duplication and speeds up future module additions.

Recommended API functions:
- getHealth()
- getVersion()
- saveToken(payload)
- getToken(clientId, platform)
- saveAccount(payload)
- getAccount(id)

Each function should:
- Call backend endpoint
- Parse JSON safely
- Return normalized success shape
- Throw normalized error shape: { status, message, raw }
